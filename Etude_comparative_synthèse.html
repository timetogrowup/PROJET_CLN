Présentation du Projet CLN

Explication technique des solutions

Solution 1 - Pipeline de Données

Le pipeline de données extrait, transforme, et charge des données de différentes sources vers un entrepôt de données ou un autre système cible. Cette approche repose sur un processus ETL (Extraction, Transformation, Chargement) pour traiter de grandes quantités de données et les rendre prêtes à l'utilisation.

  • ETL : Processus pour extraire, transformer, et charger des données.
  • Orchestration : Utilisation d'outils comme Apache Airflow ou Talend pour automatiser et orchestrer les étapes du pipeline.
  • Automatisation : Le pipeline est souvent exécuté à intervalles réguliers ou en fonction d'événements.

Solution 2 - Architecture Microservices ou SOA

Dans une architecture microservices, chaque fonctionnalité de l'application est décomposée en petits services indépendants, qui communiquent entre eux via des API. Chaque service peut avoir sa propre base de données, son propre cycle de développement et de déploiement, offrant une flexibilité accrue.

  • API REST ou gRPC : Les microservices communiquent via des API exposées, souvent basées sur des protocoles REST ou gRPC.
  • Conteneurisation : Utilisation de Docker ou Kubernetes pour déployer et gérer les microservices dans des conteneurs.
  • Orchestration : Kubernetes gère l'automatisation du déploiement, la mise à l'échelle et la résilience des microservices.

Solution 3 - Intégration directe via API

Cette solution consiste à utiliser des API pour connecter directement plusieurs systèmes ou applications. L'API sert d'interface permettant aux différents systèmes de communiquer et d'échanger des données en temps réel sans passer par un intermédiaire complexe.

  • API REST : Les systèmes communiquent via des requêtes HTTP pour envoyer et recevoir des données en temps réel.
  • API Gateway : Un point d'entrée unique pour gérer les appels API, offrant des fonctionnalités telles que la gestion du trafic.
  • Sécurité : Utilisation d'authentifications OAuth ou JWT pour sécuriser les communications entre les applications.

Solution 4 - Application Unique Centralisée (Interne)

Cette solution centralise toutes les fonctionnalités de l'entreprise dans une seule application. Ce type d'architecture monolithique regroupe toutes les fonctionnalités sous une seule structure, ce qui la rend complexe à maintenir et à faire évoluer.

  • Monolithique : Toutes les fonctionnalités sont intégrées dans une seule base de code, avec un accès partagé à une base de données centralisée.
  • Transactions ACID : La gestion des données suit les principes de transactions ACID (Atomicité, Cohérence, Isolation, Durabilité).
  • Maintenance : Les mises à jour et modifications dans une partie de l'application peuvent impacter l'ensemble de la solution.

Solution 5 - Application Unique Centralisée (Outil du Commerce avec Spécificités)

Cette solution utilise un outil commercial (COTS - Commercial Off-The-Shelf) qui est ensuite personnalisé pour répondre aux besoins spécifiques de l'entreprise. Cela permet de déployer rapidement un produit standard tout en ajoutant des fonctionnalités sur mesure.

  • COTS : Outils tels que SAP, Salesforce, ou Microsoft Dynamics, utilisés comme base pour les processus métier.
  • Personnalisation : Ajout de modules personnalisés, intégrations, ou API pour adapter l'outil aux besoins spécifiques de l'entreprise.
  • Maintenance par l'éditeur : Les mises à jour sont gérées par l'éditeur du logiciel, mais les personnalisations peuvent compliquer l'application de ces mises à jour.
